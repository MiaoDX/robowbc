# Concerns

Generated on 2026-04-27 from the current codebase and local validation run.

This is a focused risk list, not a bug list. The items below are the main
engineering concerns that stand out from the current tree.

## 1. Complexity is concentrated in a few hotspot files

Largest maintained source files today:

- `crates/robowbc-ort/src/lib.rs` — 4,382 LOC
- `crates/robowbc-cli/src/main.rs` — 2,651 LOC
- `crates/robowbc-sim/src/transport.rs` — 1,776 LOC
- `scripts/generate_policy_showcase.py` — 2,899 LOC
- `scripts/roboharness_report.py` — 1,949 LOC

Why this matters:

- change coupling is high
- reviews become harder because logic, schema, and orchestration live together
- future refactors are more likely to disturb adjacent behavior

The current code is still understandable, but these files are the main places
where maintenance cost will keep rising if left monolithic.

## 2. Reproducibility depends on external downloads and environment state

Current examples:

- `crates/robowbc-ort/build.rs` auto-downloads ONNX Runtime 1.24.2 unless
  `ROBOWBC_ORT_DYLIB_PATH` is pre-set
- `scripts/build_site.py` auto-downloads MuJoCo 3.6.0 into
  `MUJOCO_DOWNLOAD_DIR`
- model download scripts fetch live public checkpoints used by showcase and
  benchmark flows
- CI caches model assets to keep the showcase job practical

Why this matters:

- offline or hermetic builds are harder
- failures can come from network, cache, or upstream artifact churn rather than
  the repo itself
- reproduction quality depends on environment preparation being correct

This is acceptable for a research-oriented robotics runtime, but it is still a
real operational risk.

## 3. The product surface spans Rust and Python, with some version skew

Evidence in the current tree:

- Rust-side ORT build script pins ONNX Runtime 1.24.2 shared libraries
- site Python deps pin `onnxruntime==1.24.4`
- the user-visible showcase and benchmark pages are generated by Python, not by
  the Rust runtime alone

Why this matters:

- artifact semantics and runtime semantics can drift if schemas change
- numerical or environment differences across the Rust and Python layers can be
  misread as policy differences
- changes that look “runtime local” often need cross-language verification

This is one of the main architectural realities of the repo today.

## 4. Registry linkage still depends on explicit force-link mechanics

Evidence:

- `robowbc-cli/src/main.rs` references each ORT policy type with
  `TypeId::of::<robowbc_ort::...>()`
- `crates/robowbc-py/src/lib.rs` calls `robowbc_ort::link_all_ort_policies()`

Why this matters:

- a new consumer can accidentally ship with an empty policy registry
- the failure mode is subtle because the registry code itself looks correct
- this coupling is easy to forget during new binary or binding work

The workaround is deliberate and documented in code, but it remains a sharp
edge.

## 5. Real-world validation coverage is present, but partly gated behind ignored tests

Current local state:

- `cargo test` passed with `172 passed, 12 ignored, 0 failed`
- ignored tests cover real-model inference, zenoh loopback, and optional local
  Torch paths

Why this matters:

- the default suite gives strong contract confidence
- the strongest evidence for “works with real public assets and real peers”
  still requires manual opt-in and environment prep

This is not a criticism of the current suite. It is a reminder that “green test
run” and “all realistic deployment paths exercised” are not the same thing.

## 6. Lint cleanliness relies on a non-trivial number of local suppressions

Current inventory:

- 45 `#[allow(...)]` attributes under `crates/` and `scripts/`
- notable suppressions include `clippy::too_many_lines`,
  `clippy::too_many_arguments`, and multiple cast-related exceptions

Why this matters:

- the repo is lint-clean, but some of that cleanliness is achieved by carving
  out exceptions in the hardest files
- suppressions are not inherently bad, but they often mark the same hotspots
  already carrying the most behavioral complexity

This is best read as a pressure indicator, not as a style violation.

## 7. The framework goal is broader than the currently runnable policy surface

Current public/runnable shape from the checked-in docs and configs:

- strong live support: `gear_sonic`, `decoupled_wbc`, `wbc_agile`, `bfm_zero`
- blocked or partial paths: `hover`, `wholebody_vla`
- benchmark package is explicitly NVIDIA-first
- robot and asset focus is heavily Unitree G1-centric

Why this matters:

- the long-term product claim is framework-level WBC unification
- the present runnable center of gravity is still concentrated around one robot
  family and one major upstream stack

That is consistent with the founding strategy, but it is still the main
product-scope risk if broader ecosystem adoption is the goal.

## 8. The standalone Python test suite is not obviously wired into CI by name

Evidence:

- `python3 -m pytest -q` passed locally for `tests/`
- there is no explicit `pytest` invocation in `Makefile` or
  `.github/workflows/ci.yml`
- CI does exercise Python-heavy paths through `make showcase-verify` and
  `make python-sdk-verify`

Why this matters:

- the Python/reporting layer is important enough that direct unit-test
  discoverability matters
- today, a reader has to infer coverage indirectly from the workflow commands

This is a moderate concern, not a severe one. The Python layer is being
validated, but the dedicated test suite is less obviously first-class than the
Rust suite.

## Overall Readout

The repo is healthy, but the main risks are structural rather than correctness
failures:

- a few large cross-cutting files
- environment-heavy reproducibility
- cross-language product coupling
- registry linkage sharp edges
- realistic-path validation that still depends on optional assets and peers

Those are the areas most likely to tax future iteration speed.
