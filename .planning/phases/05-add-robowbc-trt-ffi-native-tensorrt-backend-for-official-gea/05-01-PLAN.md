# Phase 5 Plan 1

**Refined objective:** make the NVIDIA comparison provider-truthful before
claiming any native TensorRT backend story. This phase deliberately stops short
of introducing `robowbc-trt-ffi`; it aligns the existing RoboWBC and official
benchmark harnesses so `cpu`, `cuda`, and `tensor_rt` rows mean the same thing
and fail loudly when they cannot be measured fairly.

## Tasks

1. Add provider label parsing and fail-fast provider session creation in
   `robowbc-ort`.
2. Replace the old ambiguous GEAR-Sonic rows with planner-only,
   encoder+decoder-only, and full live velocity cases in the Criterion bench
   and case registry.
3. Thread `--provider` through the RoboWBC and official wrappers, including
   temporary three-block provider rewrites for `configs/sonic_g1.toml` and
   provider-specific official harness setup.
4. Emit blocked artifacts when a provider or path is unavailable instead of
   silently relabeling CPU results.
5. Add regression tests for the registry, wrapper source commands, provider
   rewrites, blocked rows, and runtime failure surfaces.

## Verification

- `cargo test -p robowbc-ort`
- `python3 -m unittest tests.test_nvidia_benchmarks`
- `cargo test --workspace --all-targets`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo fmt --all -- --check`
- `cargo doc --workspace --no-deps`

## Success Criteria

- GEAR-Sonic benchmark rows are split into explicit planner-only,
  encoder+decoder-only, and full velocity-path cases.
- RoboWBC and official wrappers both accept `--provider` and record the
  requested provider in the normalized artifact.
- Unsupported provider/path combinations emit `status = "blocked"` with the
  exact blocker instead of relabeling a CPU measurement.
- End-to-end CLI cases rewrite all three GEAR-Sonic provider blocks together
  while leaving the checked-in runtime config on CPU by default.
- The benchmark registry, wrappers, and tests all agree on the new case IDs and
  source-command provenance.
