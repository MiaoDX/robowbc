### Section 2 — Error & Rescue Map

The plan now needs an explicit failure policy. Generic fallback is correct only
when no phase metadata was authored. It is the wrong behavior when phase
metadata was authored but malformed.

#### Error & Rescue Registry

| METHOD/CODEPATH | WHAT CAN GO WRONG | EXCEPTION CLASS | RESCUED? | RESCUE ACTION | USER SEES |
|-----------------|-------------------|-----------------|----------|---------------|-----------|
| CLI schedule parse | duplicate / empty / partial phase naming | `PhaseTimelineValidationError` | Y | fail config validation with segment index and field name | explicit config error |
| CLI phase timeline derivation | end tick math disagrees with requested run length | `PhaseTimelineRangeError` | Y | fail showcase config validation or emit no final published artifact | explicit build/run error |
| tracking sidecar load | sidecar missing | `SidecarNotFound` | Y | fallback to generic tracking checkpoints | generic tracking review |
| tracking sidecar load | sidecar present but malformed / overlapping / unsorted | `TrackingPhaseSidecarError` | Y | fail report generation with file path and offending field | explicit report error |
| lag capture | requested lag frame does not exist | `LagFrameUnavailable` | Y | record actual available lag options and fail validator for published showcase if required tail is missing | explicit manifest/validator failure |
| site render / bundle validate | manifest paths escape proof-pack root or labels inject unsafe markup | `PhaseManifestSecurityError` | Y | reject build, print offending path/key | explicit site-build error |

**Auto-decision:** adopt this rescue rule:

- **Fallback only when metadata is absent.**
- **Fail loudly when metadata is present but malformed.**

That keeps the reusable contract honest.
